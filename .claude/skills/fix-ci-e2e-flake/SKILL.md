# 修复 CI 偶发 e2e 失败（flake）

本仓库 e2e 用 Playwright + `_electron` 启动真实 Electron，通过 `window.__E2E__` 探针调服务。CI 偶发、本地稳过的失败**绝大多数不是产品 bug，而是断言写法不够鲁棒，或 CI 环境噪音（extension host 崩溃、进程启动慢、定时器竞态）**。核心套路：**判定真回归 vs flake → 读 call log 的“失败形态” → 把断言收敛到“被测对象本身” → 本地验证 happy path 不破 → 经验追加到案例库/速记**。

> ⚠️ 第一原则：**不要为了让 CI 变绿而削弱对被测行为的覆盖**。鲁棒化 = 排除背景噪音干扰，被测断言强度不变。只能靠放宽真正的被测断言才能过 → 它可能是真回归，别盖住。

## 判定流程

1. **真回归 vs flake**：本地 `--repeat-each=5` 能否复现（本地稳过+CI 偶发→flake；本地也挂→回归）；同 commit 重跑能过=flake，每次必挂=回归/结构性缺口；`git log -p` 看失败 spec 是否刚改过；核对 CI 堆栈绝对路径（`D:\a\...` 是 runner 路径）语义对得上你改的目录。
2. **读 call log 失败形态**：count **波动**=背景元素间歇出现（噪音污染全局 count）；count/received **稳定停错值**=被测对象自身没就位（真回归/定时器没触发/fire-once 空转）；`waiting for locator` 恒 0=渲染没发生/探针没触发/选择器错；timeout 且无元素=往前看前置步骤。
3. **已知噪音源**：extension host 偶发崩溃（`ExtensionHostClientService._handleCrash` 发背景 toast + error 日志）；进程启动慢/重启类（记忆 `e2e-relaunch-flake-windows`）；renderer 定时器竞态（auto-hide/auto-read 在 CI 晚几百 ms）。
4. **最小且鲁棒的修复**（优先对齐同文件已鲁棒化的兄弟断言）：噪音污染**列表**→`.filter({hasText:'<被测唯一文案>'})` 收敛；噪音污染**全局单值/一次性状态**→从源头禁用无关子系统（先 grep 确认无 spec 依赖）；定时器/异步竞态→`expect.poll`/`toHaveCount({timeout})`，少用固定 `waitForTimeout`+硬断言；纯环境型→别强改产品，记录案例库。
5. **验证**：`pnpm --filter @universe-editor/editor exec playwright test e2e/specs/<spec>.ts [--repeat-each=5]`；全量 `pnpm e2e`（输出多，只截错误）。本地无法复现 CI 噪音是常态——目标是“鲁棒化没破坏 happy path”。
6. **沉淀**：把“失败形态→根因→修法”追加到案例库，并新增/更新一条速记（与案例号互相引用）。这是本 skill 长期价值所在。

## 速记（判定信号速查，先扫这里再读对应案例）

1. call log count **波动**=背景噪音；count **卡死错值**=更像真回归。先分清。
2. extension host CI 偶发崩溃，污染一切对通知做全局 count 的断言——按文案过滤。
3. 鲁棒化 ≠ 放宽被测断言。别用“变绿”掩盖真回归。
4. 异步/定时器用 `toHaveCount({timeout})`/`expect.poll`，少用固定 `waitForTimeout`+硬断言。
5. 同 spec/同文件内若有步骤已“按文案过滤/已加固”、有的还在“全局 count/裸 evaluate”，后者是遗留薄弱点，优先对齐复用兄弟方法。
6. CI 堆栈路径是 runner 路径（`D:\a\...`），语义对齐即可，别因路径不同误判改错 checkout。
7. 本地无法复现 CI 噪音是常态；本地验证目标是“没破坏 happy path”。
8. 噪音污染**列表**（可 filter）→改 spec 收敛；污染**全局单值/一次性内部状态**（bell badge、`_hasRevealed`）或**产品自身一次性动作**→spec 层无法隔离，从源头禁用噪音子系统/修产品（先 grep 确认无 spec 依赖）。同一噪音源可同时打挂多个看似无关的 spec。（案例 2/11）
9. **retry 救得回=瞬时竞态（flake）；救不回=结构性问题（test 超时击穿/真回归/产物缺口）**。分类第一信号，先看这个再读 call log。
10. spec 内 poll/service retry 预算（20s、180s）远大于全局 test `timeout`（30s）时，**test 超时才是真天花板**，宽 poll 是摆设。依赖子进程冷启动的 spec 用 `test.slow()` 抬 test 超时；CI 资源紧（2 核跑 4 worker）放大冷启动，`workers/expect.timeout` 按 `process.env['CI']` 分档。（案例 3）
11. 本地手跑单 LSP spec 前确保产物链齐：`pnpm ext:build`（生成 `extensions/*/dist`）+ `extension-host/dist` + `vendor/typescript-language-server`。缺 → 符号全空 `[]`/探针方法缺失，**是产物问题不是回归**。正规跑法是根 `pnpm e2e`。
12. **本地稳过 + CI 每次必挂 + received 空 + retry 救不回 = 伪 flake**（CI 漏装运行时产物，尤其**非 pnpm workspace 的 vendor**——`pnpm install`/`pnpm build` 不碰它）。别去 spec 层加超时。鉴别捷径：**同类功能“A 能跑 B 不能”的不对称**（md LSP ✓ / ts LSP ✗）。修 CI 时 `vendor-install.mjs` 可能连带 submodule，e2e job 没 `submodules: recursive`，只精准 `npm --prefix vendor/<x> ci`。（案例 4）
13. **`expect.poll` 里“盲按按键”（`keyboard.press`）是危险范式**：按键落到**当前 DOM 焦点**，被测 UI 未就位时打进别的控件（编辑器 `TEXTAREA`），**污染被测对象自身**（改文档→破坏 fixture），且**加宽窗口无效**（received 卡初值）。正解：把按键**门控在“目标已聚焦/就位”前置**（探针读状态，如 `isReferencePeekFocused()`）。诊断：fire 后盲按几次并打印 `document.activeElement`+光标，看是否随 Enter 递增。（案例 5）
14. **报错 `page.evaluate: Execution context was destroyed, ...navigation` = harness 时序 flake，不是被测断言失败**。凡“fire 后页面会 reload/导航，再 `page.evaluate(__E2E__...)`”的 PO 步骤都要对此鲁棒（捕获→重等 `domcontentloaded`+`__E2E__`→重试）。同文件已有兄弟方法（`waitForRestored`）做过加固时必须对齐复用，别留裸 evaluate。见 `WorkbenchPO._evaluateWhenRestored`。（案例 6，并入速记 5）
15. **config 已按 `process.env['CI']` 分档 `expect.timeout` 时，spec 硬编码 `{timeout:5000}` 会把 CI 分档盖回本地值**（5000===本地默认=分档失效）。received 稳定空 `""` + 等“编辑器/Monaco 首帧就位”（冷启动敏感）→ 查这步是否钉死局部 `timeout`，删掉让它继承 config 默认。**只改触发 flake 的那个谓词**，别批量改稳过用例。与速记 10 互补（一个抬 test 级，一个让 expect 级吃 CI 分档）。（案例 7）
16. **报错 Node `fs` 的 `EBUSY/EPERM/ENOTEMPTY rmdir`、栈在 `finally`/teardown 清理步 = Windows 文件锁清理竞态**，不是被测断言失败。根因：spec 自建 tmp workspace→`openWorkspace`，`finally` 删 tmpDir 时 **Electron app 还没被 fixture 关闭**（`await use(app)` 之后才 `closeApp`），句柄/watcher 仍占目录；`force:true` 不重试 EBUSY。修法：`fs.rm` 加 `maxRetries:10, retryDelay:200`。只改触发 flake 的 spec。**注意 `ENOTEMPTY` 子型（删的目录里冒出 `<id>.json.tmp`）≠ 句柄锁，而是「持续写」竞态**：主进程 storage 的 debounced 原子写（`writeFile <id>.tmp`→rename）在删除窗口内又落了新 `.tmp`，父 `rmdir` 撞 ENOTEMPTY——**`rmSync` 内建 `maxRetries` 对此救不回**（其 readdir 已过，内建 retry 只覆盖叶子删除）。须**手写循环重试整个 `rmSync`**（`for` + `await sleep(200)`，把清理函数改 async），等 in-flight 写 settle。共享 worker fixture（`sharedApp` 的 `seedUserData`）里的清理也吃这条——`reloadWindow` **不**拆主进程 storage，故 openWorkspace 类 spec 经 reset 删 workspaces 目录时偶发。（案例 8、17）
17. **`runCommand(动作)` + 紧跟 `expect.poll(只读状态)` 的分离范式，若动作是 fire-once 且依赖异步就位的前置 → received 卡死初值 + 加宽窗口无效**。典型：`focusActiveEditorGroup` 依赖 Monaco 实例 `FileEditorRegistry.register`（model 异步加载后），`monacoEditor` toBeVisible 只是 DOM 挂载；命令在 register 前 fire→静默 no-op，poll 只读不重 fire→恒 false。鉴别：**失败在前置 setup 步 + received 稳定卡初值 + 该步“fire 一次后只 poll 读”**。修法：**把动作放进 poll 谓词**，抽 PO helper。与速记 13 同源（该重做的动作没进 poll）。（案例 9）
18. **报错 `Test timeout of 30000ms exceeded`（而非 poll 自己的 message）+ received 是初值 = test 级超时击穿**，不是被测断言失败。重 spec（自己 `electron.launch`）前置开销大（launch+firstWindow+whenRestored+openWorkspace），若 poll 窗口 `timeout` ≥ 全局 test `timeout`（30s），test 天花板先到。常见成因：**从兄弟 spec 裁剪复制时漏抄 `test.setTimeout`/更大 poll 窗口**。修法：diff 兄弟对齐。鉴别“环境慢 vs 产物缺口”：**Ubuntu CI ✓ 但 Windows CI ✗ = 纯 Windows 慢**（进程创建贵+Defender 扫 spawn），不是产物缺失（那会两端都挂，见速记 12）。（案例 10）
19. **同一 flaky spec 二进宫、形态从「`Test timeout`」变「poll 自己的窗口超时 + received 恒初值」= 上轮只治了天花板（速记 18），真根因还在，八成是产品 bug**。60s poll 对一次性动作恒 0 ≠ 等不够，而是**那次动作根本没发生且永不重试**（fire-once 依赖异步前置）。查链路：**“响应事件去操作某个异步启动的资源”时，若资源的 in-flight 启动 Promise 没被 await，启动期事件会被静默丢弃**。修法范式：**操作前先 `await Promise.allSettled([...in-flight 启动 Promise])` 再读资源句柄**；修产品+加回归单测（pending spawn mock 模拟“事件撞 in-flight 启动”），别动 spec。（案例 11）
20. **报错 `Target page/context/browser has been closed`（区别于速记 14 的 `Execution context was destroyed`）先验证是不是 main 进程真崩**：抓 main 退出码，Windows `3221225477`=`0xC0000005` 访问违例=native 段错误（无 `render/child-process-gone` 事件、无 stderr、`crashReporter.start` 一开就不复现的 heisenbug）。**定位真回归 vs 环境放大的决定性方法是三组对照**：①单实例 `--workers=1` 重复多次 ②多实例 `--workers=6` 重复多次 ③多实例+禁用嫌疑子系统。**单实例怎么跑都不崩、只有多实例崩 = 测试并发放大的（常为第三方 native 库的）跨进程竞态**，真实用户单实例永不触发；**进程内**串行化修复无效（强力负结果），产品 try/catch 接不住段错误。正解：触发该 native 路径的用例 `tag:'@serial'` 隔离到 `--workers=1`。（案例 12、16）
21. **报错「Worker teardown timeout」（而非「Test timeout」或被测断言失败）+ 所有测试 pass = worker 收尾关 app 时卡死，根因在“关 app”链路，别去 spec 层找**。两步定位：①**graceful close 为何挂**——Playwright `app.close()` 走 Electron `before-quit` 的 renderer veto 链（`confirmShutdown`→各 `onBeforeShutdown` participant），任何 participant 在 headless 弹**无人应答的模态框**就让 `app.close()` 永不 resolve；②**强杀为何不彻底**——`closeApp` 超时只 `SIGKILL` **main PID**，**Windows 杀父不杀子**，node-pty/agent/ext-host 成孤儿、占管道句柄 → worker 撞 30s teardown（posix 不复现正因此差异）。修法两处互补、均不削弱断言：**Fix A 源码侧 E2E 门控**——凡 quit-chain 上会弹“headless 无人应答 modal”的 participant 按 `isE2E`（`window[E2E_PROBE_ENABLED_KEY]===true`，复用 `windowActions.ts` 先例）短路放行，让 graceful quit+`will-quit` 正常清子进程；**Fix B harness 侧 tree-kill**——`closeApp` 超时强杀改 `taskkill /pid <pid> /T /F`（Windows，`execFileSync` 同步、try/catch 吞码），非 Windows 不变。`/T` 按 parent-PID 递归，只漏 `detached:true` 独立进程组（本仓库仅外部终端打开器）。（案例 13）
22. **`page.evaluate` 里「滚动/触发后等固定 N 帧 raf 再读布局量（scrollHeight/rect）」是危险范式**：虚拟化列表（@tanstack）经 `ResizeObserver` **异步**测量进视口的行，固定帧数在慢机/并发 CI 不够，读到「估算→实测」**过渡中的瞬时值**，污染被测对象自身。**鉴别决定性信号**：`--workers=1 --repeat-each=N` 全过、`--workers=4` 全挂（received 单调爬升到某稳定值=测量收敛过程被采样）——**并发是放大器=竞态 flake，非产品回归**（与速记 9「retry 救不回=结构性」互补：retry 同样并发故救不回，但单 worker 能稳过，所以仍是 flake）。**注意 `--repeat-each` 默认按 config `workers` 跑（本仓库 4），易误判「本地也恒挂」**；必须显式 `--workers=1` 对照。修法：把「等固定帧」改为「**等布局量连续 K 帧不变（测量收敛）再采**」的 `settle()`，测的才是用户看到的稳定值；被测断言强度不变。（案例 14）
23. **滚动恢复断言用 `frac=scrollTop/max` 当落点代理是脆弱的**：①**分母漂移**——锚点恢复保证的是 `scrollTop`（哪条消息钉在顶），但视口下方内容懒测量会让 `max=scrollHeight-clientHeight` 持续变大，`frac` 即便 `scrollTop` 不动也会缓慢漂（实测从 0.4 漂到 0.11 或 0.7），两端都会误触发 `>0.15`/`<0.85` 边界；②**RAF 过冲**——`restore` 走 RAF+600ms 逐步逼近，收敛前 frac 在顶/底间震荡（含 frac=1），**单边 `poll(readFrac).toBeGreaterThan(0.15)` 会被过冲帧 frac=1 立刻满足**而提前返回，随后 `finalFrac` 读到瞬时过冲值（`finalFrac=1` flake 根因）。**正解**：断言**锚定消息的索引**（复用产品 `captureAnchor` 语义：第一条 `bottom>viewportTop` 的 `[data-slot-key]`，解析 `m:m<idx>`），它收敛后稳定不漂、直接编码被守护行为（重置到顶 index→0 / 跳到底 index→末尾）；用**绝对中部带**而非相对 targetIndex（后者在 frac=0.5 采样、上方行未测完，本身抖动）。poll 到「**连续 K 帧落在中部带**」等掉 RAF 过冲——过冲帧落首/末行清零计数。（案例 15）
24. **同类「滚动恢复」spec 多个、有的已加 echo-settle/高度收敛门控、有的没加 → 没加的在并发负载下偶发**（速记 5 的滚动版）：虚拟模式贴底依赖 `scrollToBottomStable` 的 10 帧 RAF 上限，若 echo 流式回复仍在涌入/懒测量、`scrollHeight` 还在涨，RAF 上限会在内容稳定前耗尽、**停在离底几百 px**（distance 卡 ~342，非几千），断言「贴底 distance<30」误判。修法对齐兄弟 spec：scroll 前先 `poll(getAcpMessages().length).toBe(<full>)` 等回复落地 + 两帧 `scrollHeight` 相等的高度收敛门控；单次同步 `scrollTop=scrollHeight` 也要改 `poll` 反复贴底直到 distance<30（虚拟模式一次贴底会落短）。（案例 15）
25. **spec 用 `mkdtemp(tmpdir())` 造真 git 仓库、断言依赖「spec 路径 ⋈ git 内部路径精确匹配」→ 仅 CI Windows 确定性失败（received 恒定、retry 救不回，是伪 flake=真 bug）**。根因：CI Windows 的 `os.tmpdir()` 是 **8.3 短路径**（`C:\Users\RUNNER~1\...`，`RUNNER~1`=`runneradmin` 短名），spec 把短路径喂给 `openWorkspace`/`openFileUri`；但 `git rev-parse --show-toplevel` **总返回长规范路径**，于是 `relative(长repo.root, 短file)` 破碎（`../UNB38B~2/sample.ts`）→ `git show HEAD:<rel>` `fatal: outside repository` exit 128 → `getHeadContent` 恒 `null` → dirty-diff regions 恒空 → 依赖它的 poll 恒 false。**本地 Windows `tmpdir()` 是长形故稳过**（`mkdtemp` 目录名 >8 字符必有 8.3 短名，但只有父段是短形时才触发）。鉴别决定性信号（对齐速记 12）：**「A 能跑 B 不能」不对称**——同样 `tmpdir()+真 git` 的兄弟 spec 只验 SCM provider 数量>0（不碰 `getHeadContent`）能过，唯独依赖路径精确匹配的挂。修法 spec 层最小：`realpathSync.native(mkdtempSync(...))` 把短路径规范成长形（对本地 no-op），派生的 `filePath` 随之变长形。**复现捷径**：`GetFolder(longDir).ShortPath`（COM）取 8.3 名当 cwd 跑 `git rev-parse --show-toplevel`+`relative`，无需 CI。（案例 18）
26. **诊断探针本身别污染被测断言——扩展 `getActiveDiffContent()` 这类"被多个断言共用的既有探针"追加调试后缀（`modified + '[snap=...]'`），会让该 spec 里**每一个**用它的断言都失败，且**最早的断言先挂**，把真实失败点前移、误导定位**。铁律：诊断只走**新增的独立探针方法**（`getActiveDiffDebug()`）或 `window.__X_DBG__` 数组 + 独立 probe 读回，绝不改既有断言读的返回值。另：`console.info` 太重会改变时序（heisenbug——加了诊断竞态就不复现），优先用**轻量内存数组累积事件序列**（`(w.__DBG__??=[]).push(...)`）由 probe 一次读回，既不拖慢也能看到完整调用顺序。（案例 19）
27. **e2e agent fixture（`test-fixtures/*.cjs`）写盘到 `os.tmpdir()` 且文件名只含"agent 自颁发的 sessionId"（每个 agent 进程都从 `sd-1` 起算）→ 整机共享路径，被并发 worker / Windows teardown 残留的孤儿 agent 互相覆写**。每个 test 虽 `userDataDir` 隔离，但 tmp 写盘路径不隔离。症状：diff/tracker 类断言读到**别的实例写的内容**。修法：agent 写盘路径加**进程唯一 token**（`${process.pid}-${randomBytes(4)}`）。fixture 是源码直跑（spec 喂绝对路径），改 `.cjs` 无需 rebuild，但产品 contribution 改动要 `electron-vite build` 后 e2e 才生效。（案例 19）
28. **"迟到的文件系统事件把 diff 的 modified 侧从磁盘刷回、覆盖了未保存的 live 编辑"是真产品竞态（非 spec 问题）**。链路：agent/外部写盘 `X` → 用户在打开的源文件里编辑成 `Y`（脏，未落盘，磁盘仍 `X`）→ **迟到 fs 事件**到达 → 无条件 `readFileText`→`diff.update(_, X)` 盖回。凡"响应 fs 变更去刷新某个可能同时有 live 编辑缓冲的视图"的代码，都要先查 live model：**`MonacoModelRegistry.peek(uri)` 有值就用 `model.getValue()`（编辑器缓冲才是 modified 侧真相，clean 时它=磁盘含 SCM discard 后），只有无 live model 才 fallback 读盘**。别用 `isDirty` 判断（它由 React effect 异步更新，fs 事件到达时可能还没翻 true，留竞态窗口——用 model 值本身更鲁棒）。同源三处（`ExternalChangeWatcher._refreshChangedDiffEditors`、`SessionChangesDiffSyncContribution._sync` 均须 live-model 优先）。（案例 19）
29. **e2e 用 `dispatchEvent(new Event('scroll'))` 合成"用户滚动"是危险范式：它不带 `wheel`/`pointerdown`/`keydown`，凡产品靠这些事件区分"程序化滚动 vs 用户接管"的逻辑都识别不到 → 被当成程序滚动吞掉**。典型：markdown 预览 `useMarkdownPreviewScrollRestore` 用 `restoringRef` 布尔 guard，restore 期间 `onScroll` 里 `if(restoringRef.current) return` 忽略滚动，只有 wheel/pointer/key 触发 `stop()` 才退出 restore；测试裸 `Event('scroll')` 既不匹配退出条件、又被 guard 吞，位置存不下 → 切回 `saved` 为空 → 停在 0。**放大条件**：某提交给"首次打开预览"新增了 reveal-line restore 分支（旧代码首次打开 `saved` 空直接 return、`restoringRef` 从不置 true，故不触发），且 mermaid 等**同步渲染阻塞主线程**把 restore 结束定时器（600ms）无限期拉长，合成滚动必落窗口内。**决定性鉴别**：`--workers=1 --repeat-each=N` **确定性全挂**（非并发放大，区别于速记 22）+ `git stash` 全部本地改动、`pnpm build` 回纯 HEAD **仍挂**（证明非当前改动引入，是既有 spec 脆弱点被某提交激活）+ received 恒初值 retry 救不回（速记 9）。**修法：让测试用真实 `WheelEvent`（或 `mouse.wheel`/键盘）表达用户接管**，与真实用户滚轮/拖动条/键盘一致——断言强度不变（仍断言切回后 scrollTop 恢复到 target±40）。**产品无 bug**：真实用户滚动必带这些事件，能正常退出 restore。别去改产品用位置差异（`scrollTop` 偏离）判用户接管——会把内容懒渲染/scroll-anchoring 的**高度增长漂移**误判成用户接管、停掉本该持续 re-pin 的恢复循环（亲测打破切回恢复：2720 被顶到 3549）。（案例 20）

30. **`sharedApp`（worker 级单实例、reload 复位）里，`window.reload()` 不拆主进程状态——上个测试 `openWorkspace` 打开的文件夹在 reload 后被主进程 session-restore 重新加载，把它的 editor group（含 pinned ghost tab）灌进本测试的 active group**，污染「tab 数 / 唯一活动 editor / 无重复 tab」类断言（received 多一个来路不明的 tab，本地全量偶发、单跑稳过；因**上一个跑的 spec 是谁**而随机，故形态飘忽）。`seedUserData` 删 `workspaces/` 磁盘目录与主进程 debounced session 写**竞态**，救不回（同案例 17 的写竞态源）。修法：`resetWindow` 在 `whenRestored` 后、`clearHistory` 前**确定性 `closeWorkspace()`**（切到 no-workspace scope，`WorkspaceRestoreContribution` 无状态→`clearAll()`→单个空 group），把主进程恢复的 ghost groups 拆掉。这是「主进程状态跨测试泄漏」的通用复位点，惠及所有 `sharedApp` 系 spec。（案例 21）

31. **「关闭编辑器/文件后紧接着对同一文件做 model 相关操作」有 model-dispose 竞态**：`closeAllEditors`/关 tab 命令**同步返回**，但 React 卸载 `FileEditor`→`MonacoModelRegistry.release`→dispose **是异步的**（下一次 commit）。命令返回后 `MonacoModelRegistry.peek(uri)` 仍可能拿到**即将 dispose 的孤儿 model**。凡「按 model 是否存活分「改内存 vs 落盘」两分支」的代码（如 `FileBulkEditService.apply`），此刻走 live-model 分支只 `pushEditOperations` 改内存，随后孤儿被 dispose、改动**丢失且从未落盘**。慢机/慢 CI 上卸载先于操作完成→走磁盘分支→稳过，正是 flake 根源（磁盘是否更新取决于「model 恰好是否还活着」这个非确定条件）。**决定性信号**：全量 `--repeat-each` 偶发（~1/5），非并发放大也非纯环境；探针在「close 后、操作前」读 `peek()!==undefined`（**无扰动**，别用会掩盖竞态的 console.log/数组 push——heisenbug，速记 26）能抓到孤儿存活 + 磁盘 diskText 仍旧值。修法：对「静默文件级改写」（如 markdown update-links-on-move）从源头**保证落盘**（给 apply 加 `persistToDisk` 选项：始终 `readFileText`+`writeFile`，并 `model.setValue` 同步任何残留 live model），让结果与 model dispose 时序无关——正确修复保证磁盘终态正确，不管竞态怎么跑。（案例 22）
32. **一次「启动/性能优化」重构改变就位时序（如 workbench 先挂载、Monaco/worker/workspace-hydration 后台化）会同时放大多个看似无关 spec 的「就位前」竞态；且该提交自己往往修补了一部分探针/spec 却漏改同类的其它路径——漏改处就是回归**。CI 多个 spec 在同一提交后一起变红、形态各异（outline 符号恒空、languageId 恒 `""`、命令被拒），但 `git show <opt-commit>` 里能读到「workbench mounts before Monaco finishes loading」这类**作者自述的时序变更**，且能看到它已把某些探针从 `MonacoLoader.get()` 改成 `await ensureInitialized()`、给某些 poll 加了 timeout——**顺着「它改了哪些、漏了哪些同类」就能定位全部回归**（漏改的 outline 探针/spec、漏改的 pure-preview test 即本例）。判定：先 `git show --stat <opt-commit>` 看波及面，再逐个失败对照「同类路径它修了没」。别当独立 flake 一个个猜——它们同源。（案例 23）
33. **provider 拉取的 `Promise.resolve(x).then(...)` 缺 `.catch()` → 冷启动 reject 直接掐断 retry 链，outline/结果恒空且 retry 救不回（伪 flake=真产品 bug，速记 9/12）**。`OutlineService._recomputeSymbols` 的 retry 保活逻辑 `_maybeRetry` 只在 `.then` 回调里调用；provider（如 JSON 符号 provider 委托 Monaco JSON worker）在 worker 未 warm 时 reject → `.then` 不执行 → `_maybeRetry` 不被调用 → **retry 链彻底死掉**，outline 永停在此前 provider-missing 分支 publish 的空 `[]`（还产生 unhandled rejection）。启动优化让 Monaco/worker 就位更晚，把这个隐藏缺陷从「几乎不触发」变成「CI 冷启动常态」。**决定性信号**：received 恒空 + 180s service retry 预算都救不回（区别于速记 9 的 provider-missing——那种 retry 能救回，因为 provider-missing 分支**有**调 `_maybeRetry`）。修法：`.then().catch()` 双分支都走 `_maybeRetry` 保活，catch 里加 `_currentModel !== model` 早退与 `.then` 对齐 + 调试日志。守护测试：provider 前几次返回 `Promise.reject` 再 resolve，断言 outline 最终填充（临时删 catch 内的 `_maybeRetry` 应转红）。（案例 23）
34. **`.catch()` 接得住 reject，接不住「pull 永久 pending（既不 resolve 也不 reject）」——凡「从 then/catch 里调度下一次重试」的 retry 结构，一次 hang 的 pull 就掐死全链，比速记 33 的 reject 更隐蔽**。速记 33 补了 catch 后 outline **仍在 Ubuntu 稳定挂 `[]`**：`_maybeRetry` 只在每次 pull *settle 后* 才排下一次，若 provider 的底层 **web-worker RPC / 动态 import**（JSON 符号 provider 的 `m.json.getWorker()`→`withSyncedResources`→`findDocumentSymbols` 全是 worker RPC）在冷启动时**永久排队不 resolve**，then/catch 都不触发 → 链死。**不对称鉴别（速记 12）**：Windows 本地/CI ✓（worker 冷启动快、pull 都能 settle）/ Ubuntu headless（xvfb+2 核+2 worker 争抢）✗；TS/markdown outline 走扩展主机 LSP（有自己的取消/超时）不吃这条 worker-pending 路径，故只有 JSON outline（`jsonOutline`/`outlineKeyboard`）挂。修法：给**单次 pull 加超时竞速**（`Promise.race` timeout-reject），保证它必定 settle，让 hang 被当作「一次失败重试」而非掐死全链——不误伤 happy path（超时只多一次重试，worker 下次已 warm）。守护测试：provider 前几次返回 `new Promise(()=>{})`（永不 settle）再返回符号，断言 outline 最终填充（临时把 `PULL_TIMEOUT_MS` 调到极大等于禁用超时应转红）。（案例 24）

### 案例 3：LSP 冷启动类 spec 被 30s test 级超时击穿（速记 9/10）
- **现象**：`smoke.outline` `@p1` 硬失败（retry 救不回，超时而非拿到 `[]`）；同轮 `smoke.editor`/`editorResolver`/`peekNavigation` flaky（retry 救回）。本地稳过。

### 案例 4：outline CI 每次必挂 + 拿到空 []——CI 漏装非 workspace 的 vendor（伪 flake/产物缺口，速记 12）
- **现象**：`smoke.outline` `@p1` 在 **Windows+Ubuntu 都总挂**（retry 救不回），`expect.poll(20s)` received 恒 `[]`。本地稳过。
- **根因**：outline 是**唯一依赖 typescript LSP** 的 spec（其余走 markdown LSP）。ts 扩展激活硬依赖 `UNIVERSE_TSLS_CLI`/`TSSERVER`（指向 `vendor/typescript-language-server/node_modules/...`），该 vendor **不在 pnpm workspace 内**（npm 子项目，`node_modules` ignore），只由 `vendor-install.mjs` 装；CI e2e job 只 `pnpm install`+`pnpm build`，**从未装它** → provider 没注册，符号恒空。不对称信号：md LSP ✓ / ts LSP ✗。
- **修法**：e2e job Build 前加 `npm --prefix vendor/typescript-language-server ci`（**不能**调完整 `vendor-install.mjs`——它遍历 `claude-agent-acp` submodule，而 e2e job 没 `submodules: recursive`）。同轮 peekNavigation flake 是真竞态，poll 窗口按 CI 分档 `CI?20000:10000`。
- **锚点**：`.github/workflows/ci.yml`（e2e job）；`extensions/typescript/src/extension.ts`（缺 env 不激活）；`scripts/release/vendor-install.mjs`（`VENDOR_DIRS`）；`vendor/typescript-language-server/{package.json,package-lock.json}` 入库、`node_modules` ignore。

### 案例 5：peekNavigation 20s 窗口仍超时停在 a.md——poll-press 盲按 Enter 污染被测对象自身（速记 13）
- **现象**：`smoke.peekNavigation` `@p1`「Enter follows the reference」Ubuntu CI 偶发挂，`expect.poll` 20s 超时，received 稳定 `…/a.md`（案例 4 加宽到 20s 没根治）。本地稳过。

- **锚点**：`smoke.peekNavigation.spec.ts`；新探针 `shared/e2e/contract.ts`+`renderer/e2e/probe.ts`；产品同款选择器 `PeekNavigationContribution.ts`（`.ref-tree`）。

### 案例 6：disposableLeak whenRestored 评估时 Execution context was destroyed——重启 PO 末步裸 evaluate（速记 5/14）

### 案例 7：gotoSymbol activeEditorLanguageId poll 5s 超时 received 恒 ""——硬编码 timeout 盖掉 CI 分档（速记 15）
- **现象**：`smoke.gotoSymbol` `@p1` Windows CI 偶发挂，首步 `expect.poll(getContextKey('activeEditorLanguageId'),{timeout:5000})` received 恒 `""`。本地稳过。
- **根因**：`config` 已把 CI `expect.timeout` 分档为 10s（为 Monaco 冷启动首帧留余量），但这步**硬编码 `{timeout:5000}`** 盖回本地默认 → CI 分档失效。同型散落 4 spec。
- **修法**：删掉这 4 个 spec 里 `activeEditorLanguageId` poll 的 `{timeout:5000}`，继承 config 默认。断言不动。**只改触发 flake 的谓词**，别批量改稳过用例。

### 案例 8：editorTabDnD EBUSY rmdir in finally——Windows 文件锁清理竞态，fs.rm 缺重试（速记 16）
- **现象**：`smoke.editorTabDnD` `@p1` 本地 Windows 偶发挂，`EBUSY: ...rmdir`，栈在末尾 `finally` 的 `fs.rm(tmpDir)`；被测断言此前已过。
- **根因**：spec 自建 tmp workspace→`openWorkspace`；`finally` 删 tmpDir 时 **Electron app 还没被 fixture 关闭**（`await use(app)` 后才 `closeApp`），pinned 文件句柄+watcher 仍占目录；`force:true` 不重试 EBUSY。split/多 pinned 文件故独此偶发。
- **修法**：`fs.rm` 加 `maxRetries:10, retryDelay:200`（Node 内建 Windows 重试退避）。同型裸清理一并对齐（explorerDnD/explorerExternalWatch 等）；单文件删除（update spec）不属此竞态，不动。
- **锚点**：`smoke.editorTabDnD.spec.ts`（`finally` 的 `fs.rm`）；关闭时序 `electronApp.ts`（`await use(app)` 后才 `closeApp`）。

### 案例 9：commandPalette editorFocus poll 5s 超时 received 恒 false——fire-once focus 在 Monaco 注册前空转（速记 17）
- **现象**：`smoke.commandPalette` `@p0` CI 偶发挂，失败在**前置 setup**（`expect.poll(getContextKey('editorFocus')).toBe(true)`，紧跟 `runCommand('focusActiveEditorGroup')`），received 恒 false。本地稳过。

### 案例 24：案例 23 续集——补了 `.catch()` 后 jsonOutline/outlineKeyboard **仍在 Ubuntu 稳定挂 `[]`**，真根因是「pull 永久 pending」掐死 retry 链（`.catch()` 救不到），加单次 pull 超时竞速根治（速记 34，伪 flake 鉴别 9/12/33）
- **现象**：`smoke.jsonOutline.spec.ts:30` + `smoke.outlineKeyboard.spec.ts:40` 两个 `@p1`，`getOutlineSymbols()` 20s poll 超时 received 恒 `[]`，**非常高概率仅 Ubuntu CI**、Windows 本地+CI 不复现。案例 23 的 `.then().catch()` 已在位（commit `7291dd52`）。
- **判定**：received 恒空 + retry 救不回 = 伪 flake（速记 9/12）。案例 23 已修 reject 断链，但仍挂 → 还有一条 `.catch()` 覆盖不到的死法。审 `OutlineService` retry 结构：`_maybeRetry` **只在每次 pull *settle 后*（then/catch 回调里）** 才排下一次；`.catch()` 接得住 **reject**，接不住「pull Promise **永久 pending**（既不 resolve 也不 reject）」。JSON 符号 provider 的 pull 链 `await m.json.getWorker()`（含动态 `import('./jsonMode.js')`）→`await getWorker(uri)`（`withSyncedResources` worker RPC）→`await worker.findDocumentSymbols(...)`（worker RPC）全是 **web-worker RPC / 动态 import**，Ubuntu headless（xvfb+2 核+2 worker 争抢）冷启动时 worker 迟迟不 ready → RPC 请求永久排队 pending → then/catch 都不触发 → `_maybeRetry` 永不调 → 链死，outline 永停 provider 出现前 publish 的空 `[]`。**不对称鉴别（速记 12）**：Windows worker 冷启动快、pull 都能 settle 故稳过；TS/markdown outline 走扩展主机 LSP（自带取消/超时）不吃 worker-pending 路径，故只有 JSON outline 挂。
- **根因**：retry 链「从 pull 的 then/catch 里调度下一次」的结构，遇「pull 永久 pending」无法自愈——一次 hang 掐死全链，比案例 23 的 reject 更隐蔽。
- **修法（修产品，不削弱断言、不误伤 happy path）**：抽 `_pullWithTimeout(provider, model)`，把 `provideDocumentSymbols` **race 一个 `PULL_TIMEOUT_MS=5000` 的 timeout-reject**（手写 `new Promise` + `settled` 守卫 + `clearTimeout`），保证单次 pull 必定 settle；hang 的 pull 被超时 reject → 落 `.catch()` → `_maybeRetry` 照常排下一次（worker 下次已 warm）。超时只多一次重试，健康 pull 远快于 5s。
- **守护测试**：`OutlineService.test.ts` 新增「provider HANGS」用例——前几次 `provideDocumentSymbols` 返回 `new Promise<never>(()=>{})`（永不 settle）、之后返回符号，`advanceTimersByTimeAsync(30000)` 后断言 outline 填充。**反向验证**：`sed` 把 `PULL_TIMEOUT_MS` 改极大（等于禁用超时）该用例转红，恢复转绿。
- **验证**：`pnpm check` 全绿（356 files/3317 tests）；`electron-vite build` 后 `smoke.jsonOutline`+`smoke.outlineKeyboard --repeat-each=3` 6/6（Windows 本地无法复现 Ubuntu 冷启动竞态，目标=happy path 未破，速记 7）。
- **锚点**：`OutlineService.ts`（`_pullWithTimeout` + `PULL_TIMEOUT_MS`；`_recomputeSymbols` 改调它）；守护测试 `OutlineService.test.ts`（HANGS 用例，与案例 23 的 REJECTS 用例并列）；死链结构 `_maybeRetry`（只从 then/catch 排下一次）；JSON worker 链 `JsonLanguageFeaturesContribution._provideDocumentSymbols`→`monaco/esm/vs/language/json/{monaco.contribution,jsonMode,workerManager}.js`；关联案例 23（同源 retry 缺陷）、速记 34。
