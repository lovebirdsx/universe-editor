# 分支与合并

分支让你可以"另开一条线"去改东西，不打扰主线；改满意了再合并回来。这一页讲分支的日常操作，以及本仓库常用的进阶机制——工作树。

## 分支是什么

分支就是一条独立的改动线。想试验一个没把握的新玩法，就开一条"试验"分支去改，怎么折腾都不影响主线内容；试成了再[合并](../reference/glossary.md#提交--commit)回主线，试砸了直接丢掉分支即可。主线始终干净、随时可用。

## 分支的日常操作

分支相关命令集中在源代码管理侧栏标题栏的 `…` 菜单 →"分支"子菜单里，也都能从命令面板搜到：

| 命令 | 作用 |
| --- | --- |
| **签出到…** | 切换到某个分支（把工作目录换成那条线的内容） |
| **创建分支…** | 从当前位置新开一条分支 |
| **重命名分支…** | 给分支改名 |
| **删除分支…** | 删掉一条不再需要的分支 |

> 术语提示：切换分支这个动作，侧栏里叫**签出**。在 [Git 图谱](./git-graph.md) 里，同一个动作被叫作**检出**。两者是一回事，只是界面用词不同，见[术语表](../reference/glossary.md#工作树worktree)。

## 合并与变基

把一条分支的改动并入当前分支，有两种方式：

- **合并分支…**：把另一个分支的改动合进当前分支，历史里会留下一个"合并点"。这是**推荐给创作者的默认做法**，直观、好理解。
- **变基分支…**：把当前分支的改动"搬到"另一分支的最新位置上，历史更线性。这属于进阶操作，不熟悉时先用合并。

合并两条都动过同一处内容的分支时，可能出现**冲突**，需要你手动决定保留哪边，处理办法见 [拉取、推送、储藏与冲突](./conflicts.md)。

## 发布分支

在本地新建的分支，一开始只存在于你的电脑上。运行 **发布分支**，把这条新分支第一次推送到[远程](../reference/glossary.md#源代码管理scm)，之后队友才能看到它、你也有了云端备份。

## 工作树（worktree）

**工作树 (worktree)** 是 Git 的一个进阶特性：让同一个仓库的多个分支，各自占用一个独立文件夹、同时打开，互不打架。比如主线开发的同时，你想在另一个窗口里看/改另一条分支，用工作树就不用来回切分支。刚上手可以先略过，需要并行处理多分支时再回来看。

> **重要 · 别被界面用词绕晕**：源代码管理侧栏里把 worktree 译成了 **"工作区"**（创建工作区…/打开工作区/删除工作区…）。但本编辑器里"工作区"平时另有所指——它通常指**你打开的那个项目文件夹**。为避免混淆，本册正文一律称它为 **"工作树 (worktree)"**，并强调：**工作树不是你平时打开的项目文件夹**，而是同一仓库另一条分支的独立副本目录。在 Git 图谱里，这个概念倒是正确地译成了"工作树"。

工作树相关命令在 `…` 菜单 →"工作区"子菜单（界面用词），或命令面板：

| 命令（界面用词） | 作用 |
| --- | --- |
| **创建工作区…** | 为某分支新建一个工作树目录 |
| **打开工作区** | 打开一个已有的工作树 |
| **在新窗口中打开工作区** | 在新编辑器窗口里打开工作树 |
| **删除工作区…** | 删掉一个工作树目录 |
