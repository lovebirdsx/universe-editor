## 为什么需要版本控制

如果你玩过带存档的游戏，你已经理解了版本控制的核心。把它想成给你的项目开了"随时存档、随时读档"的能力：

- **存档**：每次你觉得"这一版可以了"，就记录一个存档点。项目此刻完整的样子被保存下来，永远回得去。
- **回退**：改砸了、思路走偏了，一键读档，回到上一个满意的存档点，不用手忙脚乱地手动撤销。
- **协作**：几个人同时改同一套内容时，版本控制帮你把各自的改动合到一起，而不是互相覆盖、谁存谁的算数。

这三件事——存档、回退、协作——就是你用版本控制的全部理由。下面这些名词，都是围绕它们展开的。

## 先认识源代码管理侧栏

版本控制的主战场，是主侧边栏那一列图标里的第三个（图标是一个分叉的节点）。点开它就是**源代码管理侧栏**。

> 界面用词说明：活动栏上这个容器的标签是英文缩写 **"SCM"**，但点开后视图标题是中文 **"源代码管理"**。本册统一称它为"源代码管理侧栏"。

打开后，自上而下你会看到：

1. **提交信息输入框**：在这里写一句话，说明你这次改了什么。
2. **提交按钮**：写好说明后，点它把改动记成一个存档点。
3. **"暂存的更改" / "更改" 两个分组**：下半部分列出你动过的文件。"更改"是你改了但还没挑出来的文件，"暂存的更改"是你已经挑好、下次存档要带上的文件。

<!-- 截图：源代码管理侧栏全貌（输入框 + 暂存的更改/更改分组） -->

> 如果你的项目里含子模块（submodule），源代码管理侧栏会为每个仓库分别显示一套输入框和分组，各管各的。刚上手时不用管这个。

想让侧栏刷新到最新状态，可以运行命令 **刷新**（快捷键 `Ctrl+Alt+G`）。

## Git 概念最小集

只要记住下面六个词，日常使用就够了。每个词一句话：

- **仓库**：被版本控制管着的项目文件夹——一个装了"记账本"的项目。你打开的项目要先是仓库，才有版本控制。
- **改动**：你对文件做的、还没存档的修改。
- **[暂存](../reference/glossary.md#暂存--stage)**：从一堆改动里，挑出这次要存档的那些，放进"暂存的更改"。
- **[提交](../reference/glossary.md#提交--commit)**：把暂存的改动记成一个存档点，并附上一句说明。一次提交 = 一个存档点。
- **分支**：一条独立的改动线。比如你想试验新玩法，就开一条"试验"分支去改，不影响主线；满意了再合并回来。
- **远程**：云端或服务器上的仓库副本，用来备份和多人协作。你把提交"推"上去，别人就能拿到。

> 你不需要一次记住全部。真正高频的只有**改动 → 暂存 → 提交**这条主线，见 [提交你的改动](./commit.md)。

## 让你的项目变成仓库

版本控制只对**仓库**生效。如果你打开的项目文件夹还不是 Git 仓库：

- 源代码管理侧栏会是空的；
- [Git 图谱](./git-graph.md)会提示"Git 图谱不可用 - 此文件夹是 git 仓库吗？"。

本编辑器目前**没有内置"初始化仓库"的命令**。要让项目变成仓库，有两条常规路径：

- 用系统安装的 Git，在项目目录里执行 `git init`（把现有文件夹变成新仓库）；
- 或者**克隆**一个已经存在的远程仓库到本地（连内容一起拿下来）。

任选一种，让项目成为仓库后，回到编辑器，源代码管理侧栏和 Git 图谱就会亮起来。
