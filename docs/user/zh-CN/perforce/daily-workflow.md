## 面板长什么样

Perforce 面板按 **changelist** 把你签出的文件分组：

- **待收集的改动** 组固定在最上（有内容时才显示）：你在磁盘上改动过、但**还没签出**的文件——参见下一节。
- **默认 changelist** 紧随其后，输入框里的文字就是它提交时的描述。
- 每个**编号 changelist** 一组，组名带上它的描述。
- 若某个 changelist 有搁置内容，它以子组的形式**嵌套在该 changelist 组下方**（与 p4v 一致）。

每个文件行右侧的字母标记它的动作：`edit` / `add` / `delete` / `S`（shelved）等。

> **长操作有进度提示**：submit、shelve、还原、删除 changelist 等可能较慢的 p4 操作执行期间，状态栏的 Perforce 条目会显示带旋转图标的进度文案（如 `client: submit…`），完成后自动恢复常态，让你清楚知道后台正在跑什么。

## 收集改动（reconcile）

和 Git 直接读磁盘不同，Perforce 面板默认只显示**已签出**的文件。如果你在编辑器外（或关闭自动签出时）改了、建了、删了文件，它们不会自动出现——需要先"收集"进来。

**「待收集的改动」组**就是干这个的，对标 Git 的未跟踪 / 已修改列表：

1. 点面板标题栏刷新旁的下拉菜单，选 **刷新（清理）**，或命令面板搜 "Perforce: 收集改动 / 收集全部改动"。集成会跑一次 `p4 reconcile -n`（**只扫描、不改动服务器**），把偏离仓库的文件列进该组：
   - 磁盘上改过但没签出 → 推断为 `edit`
   - 新建、仓库里还没有 → 推断为 `add`
   - 磁盘上删了、仓库里还在 → 推断为 `delete`
2. 单击文件行看它与 have 版本的差异。
3. 在文件行上点 **收集改动**（`+`），或在组标题上点 **收集全部改动**，集成对这些文件跑真正的 `p4 reconcile`，把它们签出到对应的 changelist——文件随即离开"待收集"组，出现在下方的 changelist 组里，就能提交了。
4. **不想要某项改动**？在「待收集的改动」组里的文件行或文件夹行上右键选 **还原修改**（会先确认），把该文件恢复到 have 版本——磁盘上改过的内容被丢弃、新增的文件被删除、被删的文件被找回。这走的是 `p4 clean`，专门清理尚未签出的偏离，是不签出、直接丢弃磁盘改动的快捷方式。
5. **只想让它别出现在列表里、但保留磁盘改动**？在文件行 / 文件夹行 / 组标题上选 **从列表移除**（`eye-off` 图标）。它**不动磁盘、不动服务器**，只是把这些条目永久隐藏——即便之后再改动该文件、或做**刷新（清理）**全量扫描，也不会再冒出来。适合把一批你明确不打算提交的本地改动（临时脚本、本地配置）从视野里清掉。想让它们重新出现，在组标题上选 **显示已移除的改动**（清空移除记录并重新全量扫描）；或直接对该文件**收集改动**，收集会自动解除它的移除标记。

> **列表会记住，也会自愈。** 「待收集的改动」列表和「已移除」记录都按工作区持久化：重开编辑器会**立即恢复上次的列表**，不重新全量扫描（大型工作区因此秒开）。之后每次普通刷新都会顺带核对列表里现有条目的磁盘状态——某个改动若已被丢弃（内容改回 have 版本、新增的文件被删、被删的文件被找回），会**自动移出列表**，无需手动全量扫描。想要一次权威的重新发现，用**刷新（清理）**。

> **想让它一直自动扫描？** 开启 `perforce.autoReconcile`（默认关）：此后每次刷新都会顺带扫描待收集的改动。默认关闭是因为大型工作区扫描（`p4 reconcile -n`）可能较慢；平时用「刷新（清理）」或「收集改动」按需触发即可。

## 签出、新增、删除

Perforce 里改文件前要先"签出"。三种改动方式：

- **签出（edit）**：把已在仓库里的文件标记为可改。
- **新增（add）**：把新文件纳入版本控制。
- **删除（delete）**：把文件标记为删除（会有确认提示）。

有三处入口：

- **命令面板**（`Ctrl+Shift+P`）搜索 "Perforce: 签出 / 新增 / 删除"。
- **资源管理器右键**：在文件树里对某个文件直接选 "Perforce: 签出 / 新增 / 删除 / 打开更改 / 收集改动"；对**目录**右键则可选 "收集改动"，递归收集整个子目录的改动。
- **面板文件行 / 组**上用对应动作。

> 这些右键 / 标题栏入口只在文件归属某个 Perforce 工作区时出现——打开非 Perforce 仓库时不会显示 Perforce 菜单项。

## 自动签出

不想每次手动签出，可开启 `perforce.autoEdit`（默认关）。开启后，当你**编辑一个属于当前工作区、尚未签出的文件**时，集成会自动对它跑 `p4 edit`。这与许多团队的 Perforce 使用习惯一致。

> 自动签出只在你真正修改文件内容时触发，不会因为只是打开文件就签出。

## 对比与打开改动

- 在面板里**单击**文件行：打开本地版本与仓库 **have 版本**的差异对比。
- **打开当前文件的改动**：对活动编辑器里的文件直接开差异视图——点编辑器标题栏的对比图标，或命令面板搜索 "打开更改"。

编辑器左侧的**改动色条（dirty-diff）**同样对 Perforce 文件生效：改动的行会显示色条，点色条弹出行内小 diff。详见 [Git 的改动色条说明](../git/session-changes.md)，Perforce 行为一致。

## 还原

**还原（revert）**丢弃对文件的签出改动，把它恢复到 have 版本（会有确认提示）。

- **单个文件**：在文件行上用还原动作。
- **整组还原**：在默认 / 编号 changelist 组标题上用 **还原 Changelist 中的所有文件**，一次丢弃该组全部签出改动（会有确认提示）。
- **还原未改动文件（revert unchanged）**：只还原那些被签出但内容其实没变的文件，清理噪音很实用（安全，不会丢真实改动，无需确认）。

## 提交

1. 在默认 changelist 的输入框里填提交描述。
2. 点输入框旁的 **提交（submit）** 按钮，或命令面板搜 "Perforce: 提交默认 changelist"。
3. 也可以直接**提交某个编号 changelist**（在该组上用提交动作）。

> **默认动作随 Swarm 而变**：当 [Swarm 审核](./swarm-code-review.md)已开启且配置了服务器地址时，输入框旁按钮的**默认动作会变为「发起新的 Swarm 审核…」**——契合"改动先审核、再提交"的团队流程；点按钮旁的下拉箭头仍可切回直接提交。未启用或未配置 Swarm 时，默认动作保持为提交。

> 提交会直达中央 depot，**不可撤销**（不像 Git 有本地 amend / undo）。因此每次提交都会先弹确认框。提交成功后面板自动刷新。

## 刷新

Perforce 状态在服务器上，服务器本身没有文件监视器。集成在每次操作后自动刷新；你也可以：

- 用面板标题栏的**刷新**按钮手动刷新。刷新期间按钮会变为转圈加载态并暂时禁用，当前刷新完成后才恢复可点，避免重复触发。
- 用刷新旁下拉菜单里的**刷新（清理）**：在普通刷新之外，额外扫描[待收集的改动](#收集改动reconcile)。
- **文件监视自动刷新**（`perforce.autoRefresh`，默认开）：监视工作区磁盘，无论是编辑器内保存还是外部工具的改动，只要文件变了就自动刷新，并**只针对刚变动的那些文件**扫描待收集的改动（而非重扫整个目录），因此在超大型目录下也很快。这样改完文件就能在面板里看到，无需手动签出。想一次性扫出目录里所有历史改动，用**刷新（清理）**做全量发现。
- 开启**轮询**：把 `perforce.refreshInterval` 设为秒数（最小 10 秒，`0` 或留空为关闭），集成会定时刷新。详见[冲突解决与进阶设置](./resolve-and-advanced.md)。
